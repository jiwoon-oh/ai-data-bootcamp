# 모두마켓 월간 로그 — 도구 선택 보고서

## 1. 데이터 개요
- 파일: web_logs.csv
- 크기: 32.8 MB, 500,000 행
- 주요 컬럼: ts / user_id / page / device / status_code / response_ms / bytes_sent

## 2. 측정 결과
| 방식 | 소요 시간 | 메모리 피크 |
|---|---|---|
| pandas + dtype | 1.41초 | 20.0MB |
| pandas chunked(10K) | 2.51초 | -2.7MB | (노이즈가 섞였을 가능성 있음)
| pandas chunked(50K) | 1.64초 | 1.1MB |
| pandas chunked(100K) | 1.81초 | 4.4MB |
| pandas chunked(200K) | 1.49초 | 6.3MB |
| Polars lazy    | 0.21초 | 97.6MB |

## 3. 분석 결과 요약
- 페이지별 평균 응답 시간 최고: search 페이지 (160.42ms)
- 디바이스 점유율: mobile 69.9%, desktop 25.1%, tablet 4.9%
- 에러 1위 페이지: home (42,898건)

## 4. 도구 선택 정당화 (한 단락)
이 분석에는 **Pandas Chunked(50K)** 을 선택했습니다. 이유는
측정 결과 Polars lazy방식이 가장 압도적으로 빠르긴 하지만 Pandas chunked방식이 메모리 관련해서 1.1MB로 가장 많이 절약을 할 수 있었습니다. 또한 팀 사정상 아직 Pandas를 쓰는 팀원들뿐이라 Pandas + Chunked 방식으로 불필요한 데이터만 걷어내는 방식을 선택하였습니다.

## 5. Chunksize 50K 고정 이유
- 10K, 50K, 100K, 200K 모두 비교해보았다
청크는 작게 쪼갤수록 오버헤드가 여러번 반복되어 시간이 늘어나고 청크를 크게 잡을수록 오버헤드 횟수는 줄지만 한번에 메모리에 올리는 양이 늘어난다.
- 즉 청크가 너무 적어도 안되고 너무 많아도 오히려 메모리부담이 더 커질 수 있다, 따라서 전체 행의 10%정도만 청크로 잡는것이 가장 최적화가 되어있다고 할 수 있다.
